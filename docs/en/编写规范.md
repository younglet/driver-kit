# Hardware Driver Documentation Guide & Category Master

> This file is the authoring guide, structural template, and category plan for all hardware experience documents.
>
> One Markdown file per hardware, using a unified filename:
>
> ```text
> category-field-module.md
> ```
>
> Example:
>
> ```text
> 传感器-位姿-三轴陀螺仪.md
> ```

## 1. Document Organization Principles

### 1.1 One file per hardware type

The following content for the same hardware type MUST be in a single file:

- Function and operating principle
- Application scenarios
- Common part numbers
- Key parameters
- Reference prices
- Selection guide
- Hardware connections
- Initialization flow
- Driver API
- Data format and units
- Calibration, filtering, and compensation
- Error handling
- Debugging methods
- Example code

Do not split the same hardware into a "description file" and a "driver file".

### 1.2 Index file handles navigation only

`index.md` only:

- Lists all documents
- Organizes links by category and field
- Provides quick navigation
- Does NOT duplicate module content
- Does NOT maintain part numbers, prices, or driver notes

### 1.3 This file is the writing guide & plan

`总表-硬件驱动文档分类.md` handles:

- Explaining the file naming convention
- Defining the section structure for individual module documents
- Defining content quality requirements
- Maintaining the hardware category master table
- Marking modules as completed or planned

Hardware-specific content MUST live in its corresponding module file.

## 2. Standard Document Structure

Each hardware document should follow this section plan:

```markdown
# Module Name

## 1. Module Overview
## 2. Working Principle and Use Cases
## 3. Common Part Numbers and Reference Prices
## 4. Key Parameters and Selection Guide
## 5. Hardware Connection and Electrical Notes
## 6. Communication Method or Control Signal
## 7. Initialization Flow
## 8. Common Driver Interfaces
## 9. Data Format, Units, and Timing
## 10. Configuration Parameters
## 11. Calibration, Compensation, and Filtering
## 12. Interrupts, DMA, FIFO, and Buffers
## 13. Error Handling and Exception Recovery
## 14. Low Power
## 15. Example Code
## 16. Debugging Methods
## 17. Frequently Asked Questions
## 18. Development Notes
## 19. References
```

Sections that don't apply may be removed; avoid padding irrelevant content just to fit the template.

## 3. Section Requirements

### 3.1 Module Overview

State what the module measures or controls, and whether it falls under sensors, actuators, communication interfaces, or other hardware.

### 3.2 Working Principle and Use Cases

Briefly describe how it works, and list common applications. Do not paste entire datasheet sections into the document.

### 3.3 Common Part Numbers and Reference Prices

Use a table like:

| Part Number | Type | Interface | Highlights | Module Price | Bare Chip Price |
|---|---|---|---|---:|---:|
| Example | Module / Bare chip | I2C / SPI | Brief highlights | $5~15 | $3~10 |

Price requirements:

- Use USD ($) estimates with a range (international retail reference)
- Note whether the price covers a bare chip, module, or development board
- Explain that prices vary by channel, brand, batch, and time
- Do not state estimates as guaranteed prices
- Formal procurement MUST use vendor quotes and official datasheets

### 3.4 Key Parameters and Selection Guide

At minimum, list parameters directly relevant to the use case:

- Supply voltage
- IO logic level
- Measurement range
- Resolution
- Accuracy and noise
- Sampling rate
- Power consumption
- Temperature range
- Interface
- Package
- Supply chain status

### 3.5 Hardware Connection and Electrical Notes

Must address:

- Power and ground
- Signal levels
- Pull-up, pull-down, or termination resistors
- Chip-select, enable, and interrupt pins
- External capacitors or protection devices
- Voltage, current, and timing limits that must not be exceeded

### 3.6 Communication Method or Control Signal

Describe the communication protocol, pin functions, device address, chip-select, baud rate, clock mode, or PWM / pulse parameters.

### 3.7 Initialization Flow

Initialization should cover:

1. Configure the underlying bus
2. Check whether the device is present
3. Read device ID or revision
4. Reset the device
5. Set operating mode
6. Set key parameters
7. Clear status, FIFO, and interrupts
8. Perform necessary self-tests

### 3.8 Common Driver Interfaces

Distinguish between lifecycle, data, configuration, and diagnostic interfaces:

```text
begin()
reset()
sleep()
wake()
read()
read_raw()
read_status()
data_ready()
set_range()
set_rate()
set_mode()
set_filter()
calibrate()
self_test()
is_connected()
```

Specific names may be adjusted per platform, but stay consistent within a project.

### 3.9 Data Format, Units, and Timing

Must specify:

- Return value meaning
- Units
- Signed or unsigned
- Byte order
- Scale factors
- Data update time
- Maximum read interval
- Timeout conditions

### 3.10 Calibration, Compensation, and Filtering

Describe calibration conditions, sample count, persistence, temperature effects, filter latency, and outlier handling. Calibration results must consider versioning, checksums, and a way to restore defaults.

### 3.11 Interrupts, DMA, FIFO, and Buffers

Describe buffer capacity, overflow flags, clear methods, frame length, timestamps, and concurrent-access rules.

### 3.12 Error Handling and Exception Recovery

Cover at least:

- Device not responding
- Communication timeout
- Checksum failure
- Data not ready
- Buffer overflow
- Out-of-range values
- Device power-loss or reset
- Bus busy or stuck

Provide strategies for retry, reset, re-initialization, and fault reporting.

### 3.13 Low Power

Describe power consumption in active, standby, sleep, and wake states. Clarify wake-up sources and what must be reconfigured after wake-up.

### 3.14 Example Code

Example code must:

- Demonstrate the complete initialization flow
- Check initialization and communication errors
- Indicate units
- Avoid using non-existent APIs
- Note the platform or library dependencies
- Highlight where hardware-specific parameters should be modified

## 4. Document Quality Checklist

Before submitting a module document, verify:

- [ ] Filename matches `category-field-module.md`
- [ ] No splitting of one hardware into multiple files
- [ ] Common part numbers listed
- [ ] Reference price ranges listed
- [ ] Price uncertainty noted
- [ ] Supply and IO logic levels specified
- [ ] Interface, address, or control timing specified
- [ ] Initialization and device detection specified
- [ ] Units, byte order, and data format specified
- [ ] Calibration or parameter configuration specified
- [ ] Communication failure and exception recovery specified
- [ ] Minimal runnable example provided
- [ ] Original datasheet noted as the final authority
- [ ] `index.md` updated with the new file's link
